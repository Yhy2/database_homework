## 2. 项目概述

本项目实现了一个面向数据库课程作业的校园二手交易平台。系统采用 `Vue 3 + Vite + Element Plus` 构建前端页面，采用 `Flask + PyMySQL + MySQL 8` 实现后端 API 与数据库访问，所有核心业务逻辑都通过原生 SQL 完成，避免 ORM 隐藏作业重点。

页面覆盖首页、商品列表、用户列表、订单列表、查询统计页和购买演示页。前端展示的数据全部来自数据库真实结果，支持公网演示、截图、录屏和本地验收。

## 3. 运行步骤

### 3.1 本地一键运行（推荐）

1. 保持项目根目录下存在 `docker-compose.yml`、`Dockerfile` 和 `sql/` 初始化脚本。
2. 如需自定义端口或密码，可先复制 `.env.example` 为 `.env` 并修改变量。
3. 在项目根目录运行：

```bash
docker compose up --build -d
```

4. 打开浏览器访问 `http://localhost:8080`。

### 3.2 开发联调

后端开发：

```bash
uv venv .venv
uv pip install --python .venv/bin/python -r backend/requirements.txt
.venv/bin/python backend/app.py
```

前端开发：

```bash
cd frontend
npm install
npm run dev -- --host
```

开发态下，Vite 会将 `/api` 请求代理到本地 Flask 服务。

### 3.3 公网部署与验证

生产环境使用 `docker-compose.prod.yml` 启动，前端构建产物由 Flask/Gunicorn 服务统一托管，MySQL 仅在 Docker 内部网络中访问，不对公网暴露数据库端口。

生产环境启动命令：

```bash
docker compose -f docker-compose.prod.yml up --build -d
```

公网健康检查地址：

```text
http://campusmarketapply.site/api/health
```

当前健康检查返回 `status = ok`、`database = ok`，说明后端服务和数据库连接均正常。

## 4. 数据库设计与约束

系统包含 3 张核心表：

- `user`：保存用户编号、用户名和手机号。
- `item`：保存商品编号、商品名称、分类、价格、状态和卖家编号。
- `orders`：保存订单编号、商品编号、买家编号和下单日期。

关键约束如下：

- `user.user_id`、`item.item_id`、`orders.order_id` 都是主键。
- `item.seller_id` 外键引用 `user.user_id`。
- `orders.buyer_id` 外键引用 `user.user_id`。
- `orders.item_id` 外键引用 `item.item_id`，并设置唯一约束，保证每个商品最多成交一次。
- `item.status` 限制为 `0`（未售出）或 `1`（已售出）。
- `price >= 0`，防止出现非法价格。

初始化脚本位置：

- `sql/01_schema.sql`：建表与完整性约束。
- `sql/02_seed.sql`：题目初始数据与 1 条自定义商品数据。
- `sql/03_views.sql`：已售商品视图与未售商品视图。
- `sql/04_queries.sql`：作业要求中的基本查询、连接查询与聚合查询归档。
- `sql/05_purchase.sql`：购买事务示意 SQL。

## 5. 已实现的作业功能

### 5.1 数据操作

- 插入新商品：商品页支持新增商品弹窗，提交后通过 API 写入数据库。
- 修改商品价格：仅允许修改未售商品价格。
- 删除商品：仅允许删除未售商品，已售商品按钮禁用。

### 5.2 基本查询

查询统计页展示以下结果：

- 所有未售商品。
- 价格大于 30 的商品。
- `DailyGoods`（页面映射为「生活用品」）类商品。
- `u001` 发布的所有商品。

### 5.3 连接查询

- 已售商品及买家姓名。
- 每个订单的商品名、买家名和日期。
- 卖家是 `u001` 的商品是否被购买。

### 5.4 聚合与分组

- 商品总数。
- 每类商品数量。
- 所有商品平均价格。
- 发布商品数量最多的用户。

### 5.5 视图

- `sold_item_view`：已售商品视图，包含商品名和买家编号。
- `unsold_item_view`：未售商品视图。

### 5.6 业务逻辑

购买接口 `/api/purchase` 通过事务实现以下两步：

1. 在 `orders` 表中新增一条订单。
2. 将 `item.status` 更新为 `1`。

同时，后端在事务中使用 `SELECT ... FOR UPDATE` 锁定目标商品，并依赖 `orders.item_id` 唯一约束阻止重复购买。

## 6. 安全性说明

本项目实现了基础注册登录和写操作保护，同时保留一键演示登录入口：

- 普通查询接口默认开放，首页、列表页和统计页都只读访问。
- 用户可注册账号、使用账号密码登录，也可点击一键演示登录使用内置账号。
- 新增、改价、删除和购买接口都要求携带 `Authorization: Bearer <token>` 头，避免被普通访客直接误触发。
- 后端仅暴露固定 API，不提供任意 SQL 执行入口。
- 部署时关闭 `Flask debug`，避免把堆栈信息直接暴露给访问者。

如果后续要升级为真实线上系统，建议增加角色权限控制、CSRF 防护和审计日志。

## 7. 并发与恢复说明

### 7.1 并发问题

如果两个用户同时购买同一件未售商品，可能出现典型的竞争条件：两个请求都先读到 `status = 0`，随后分别尝试下单，导致重复成交。

### 7.2 解决方式

系统在购买事务中采用以下策略：

- 先执行 `SELECT ... FOR UPDATE` 锁定目标商品行。
- 在同一事务中写入订单并更新商品状态。
- 使用 `orders.item_id` 唯一约束作为第二层保护。

这样可以保证同一商品在并发场景下只有一个事务成功提交。

### 7.3 崩溃恢复

MySQL 的 InnoDB 引擎会使用事务日志（redo log / undo log）保证事务的原子性和持久性：

- 如果服务在事务提交前崩溃，未提交的修改会回滚。
- 如果服务在提交后崩溃，重启后可通过日志恢复已提交的数据。
- 实际部署时还应定期做数据库备份，避免宿主机故障导致数据永久丢失。
