# 校园二手交易平台企业官网化前端改版设计

## 1. 背景与目标

当前系统已经具备完整的数据库作业功能，但前端整体偏向「数据后台」视觉语言：功能清晰，展示完整，但品牌感、首屏表现力和页面节奏不足。为了提升演示效果，需要把整站改造成更接近大型企业官网的视觉气质，同时保留数据库作业所需的操作效率与表格可读性。

本次改版的目标不是复制某个具体品牌官网，而是借鉴其视觉语言：

- 强调企业感、秩序感和留白。
- 使用白色、浅灰、深石墨与少量红色点缀形成主视觉。
- 让首页更像官网入口页，让业务页更像「企业解决方案页面 + 业务操作台」的混合形态。
- 保留现有后端接口、数据流和主要页面结构，避免把数据库作业系统改成纯展示站。

## 2. 范围与边界

### 2.1 本次改版范围

- 重做全局导航与页面骨架。
- 重做首页、商品页、用户页、订单页、统计页、购买页的视觉结构。
- 重做全局样式 token、排版、配色、按钮、容器和表格外层组件。
- 保持现有 Vue 路由、API 封装、后端接口和数据库逻辑不变。

### 2.2 明确不做的事项

- 不改动 Flask 后端逻辑。
- 不改动 SQL、测试逻辑和业务规则。
- 不接入新的设计系统或图形资源库。
- 不逐像素模仿华为官网，不直接复刻其图片、文案或品牌资产。

## 3. 设计原则

### 3.1 官网感优先，但不能牺牲业务可读性

业务页面仍然需要承载表格、按钮、弹窗和数据查询结果。因此改版策略是：

- 用官网化的头图、模块区、卡片容器和排版建立气质。
- 用克制的表格容器与层次控制保留信息密度。
- 让用户第一眼感受到「企业官网级别的系统界面」，而不是传统管理后台。

### 3.2 全站统一，但每页要有独立角色

- 首页承担「系统形象页」角色。
- 商品、用户、订单页承担「业务管理页」角色。
- 统计页承担「数据洞察页」角色。
- 购买页承担「事务演示页」角色。

### 3.3 少量红色强调，不做大面积高饱和

红色只用于：

- 当前导航高亮。
- 主按钮与关键 CTA。
- 分区细节线和少量数字强调。

其余区域以白、浅灰、深石墨为主，形成更稳的企业感。

## 4. 视觉系统

### 4.1 配色

建议建立新的全局变量：

- 背景主色：极浅灰白。
- 内容面板：纯白或带极轻微透明度的白色。
- 主文字：深石墨黑。
- 辅助文字：中灰。
- 强调色：企业红。
- 分隔线：浅灰。
- 状态色：成功、警告、危险保持克制，不使用高饱和大面积背景。

### 4.2 排版

- 去掉当前偏生活化、暖色系的字体气质。
- 统一为更冷静、简洁的无衬线栈。
- 首页主标题使用大号、较轻但清晰的粗细。
- 页面标题、模块标题、表格标题分三级控制。
- 所有中英文与数字排版遵循中文技术文档混排规范。

### 4.3 间距与容器

- 页面采用更大的横向留白。
- 顶部导航固定悬浮。
- 每个页面至少由「横幅区 + 内容区」两层组成。
- 数据表格必须放入白色容器卡片中，避免直接裸露在背景上。

### 4.4 动效

仅保留少量动效：

- 顶栏滚动时的轻微模糊与阴影变化。
- 首页 Hero 和卡片的渐入。
- 卡片 hover 的轻微上浮。

不添加复杂轮播、视频背景或重动画。

## 5. 信息架构与页面设计

### 5.1 全局导航与骨架

`AppShell` 将从当前简单导航条改为企业官网式顶栏：

- 左侧：品牌名、英文辅助标识、简短定位语。
- 右侧：六个主导航入口。
- 当前页：红色高亮文字或红色底线。
- 主体：统一包裹到更宽的页面容器中，首屏与内容区分层清晰。

同时新增：

- 全局页面内边距规则。
- 统一页面横幅组件样式。
- 全局背景层。

### 5.2 首页

首页将从「指标 + 快捷入口」升级为官网式落地页：

- 第一屏：
  - 左侧大标题、副标题、价值陈述。
  - 右侧显示系统状态与 2 到 3 个高价值指标。
- 第二屏：
  - 以大卡片方式展示「商品管理」「查询统计」「事务购买」等核心入口。
- 第三屏：
  - 展示数据库作业覆盖能力，如 CRUD、连接查询、聚合统计、视图、事务控制。

目标是让首页在录屏和截图时具备明显的「门户感」。

### 5.3 商品页

商品页将改造成「业务管理页」：

- 顶部横幅区：标题、说明、商品总数/在售数等摘要指标、主按钮「新增商品」。
- 主内容区：表格卡片。
- 弹窗与改价区域保持现有交互逻辑，但按钮、输入区和层次会统一到新视觉系统。

重点是：保留 CRUD 效率，同时减少后台系统默认样式感。

### 5.4 用户页与订单页

这两个页面结构相对简单，但仍需拉开层次：

- 页面横幅区展示页面定位。
- 表格区域放入官网式内容容器。
- 在横幅中增加少量辅助信息或数据摘要，避免页面过空。

用户页偏「基础数据」，订单页偏「交易记录」，两者文案和强调点不同。

### 5.5 统计页

统计页会变成「数据洞察页」：

- 顶部先放关键指标卡：商品总数、平均价格、发布最多的用户等。
- 下方按模块矩阵展示基本查询、连接查询、聚合与视图结果。
- 每块结果表格都有清晰标题和说明，形成企业官网中常见的模块化陈列效果。

这页是改版中最关键的一页，因为它兼具「作业证明」和「视觉展示」双重角色。

### 5.6 购买页

购买页要强调事务逻辑而不只是按钮：

- 页面横幅区清晰说明「只允许未售商品成交一次」。
- 可在顶部增加规则说明卡片，突出事务性、一致性和重复购买限制。
- 表格下单区域保留，但按钮与买家选择区会改得更像业务操作模块。

重点是把「数据库事务演示」做得更可信、更适合录屏讲解。

## 6. 组件改造方案

### 6.1 `AppShell.vue`

- 重构为企业官网式悬浮导航。
- 品牌区和导航区视觉重构。
- 页面主容器重新定义。

### 6.2 `MetricCard.vue`

- 从当前单一指标卡升级为更强调数字与说明关系的数据卡。
- 支持更统一的色彩语义和更清晰的层次。

### 6.3 `ResultTable.vue`

- 增加更强的容器感和标题区层次。
- 调整表格外边距、标题样式、空状态文案视觉。
- 保留 `el-table` 的业务能力，不重写表格本身。

### 6.4 `ItemFormDialog.vue`

- 保留交互流程。
- 重做标题、表单节奏、按钮视觉和弹窗观感。

### 6.5 `main.css`

这是本次改版的主战场：

- 重建 token。
- 重写背景、导航、横幅、卡片、网格、表格容器、按钮、响应式规则。
- 清理现有偏暖色和偏装饰化的视觉语言。

## 7. 数据流与业务逻辑影响

本次改版不改动任何 API 协议和核心数据流：

- `frontend/src/api/*` 保持不变。
- 路由结构保持不变。
- 页面内数据加载时机原则上保持不变。
- 后端接口、SQL 查询、测试逻辑保持不变。

因此，本次改版属于纯前端表现层重构，不应影响作业结果正确性。

## 8. 响应式与兼容性要求

- 桌面端优先，适配录屏与截图场景。
- 移动端不追求完整官网式排版复现，但必须可读、可滚动、可操作。
- 窄屏下导航允许折行或转换为更紧凑布局，但不引入复杂移动菜单系统。

## 9. 风险与控制

### 风险 1：视觉过重导致业务页难用

控制方式：

- 保留 `el-table` 与现有操作路径。
- 所有大面积视觉变化放在页面壳层和模块层，不侵入业务核心交互。

### 风险 2：首页与内页风格割裂

控制方式：

- 统一导航、颜色、卡片、分区标题和按钮样式。
- 每页都使用统一的「横幅区 + 内容区」骨架。

### 风险 3：过度模仿特定品牌

控制方式：

- 只借鉴企业官网的结构和视觉节奏。
- 不使用品牌商标、原站文案或素材。

## 11. 验收标准

本次设计通过后，实施完成应满足以下标准：

- 全站视觉统一为企业官网化风格。
- 首页具备明显的首屏表现力。
- 六个页面都有统一骨架与明确页面角色。
- 表格页仍然具备良好的信息可读性与操作效率。
- 本地运行、前端构建和核心 API 交互不受影响。

## 12. 当前结论

正式采用「企业官网化的业务系统」方案：

- 全站统一改造。
- 官网感优先于现状，但不牺牲业务页可用性。
- 视觉借鉴华为官网气质，不做品牌级复刻。
